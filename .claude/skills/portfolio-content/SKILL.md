# 포트폴리오 작업물 콘텐츠 작성 스킬

## 핵심 원칙: 역할 분리

이 포트폴리오에는 **작업물**과 **인사이트** 두 가지 콘텐츠가 있고, 역할이 다르다.

| 구분 | 작업물 (Feature) | 인사이트 (Insight) |
|------|-----------------|-------------------|
| 목적 | 이 사람이 어떤 판단을 했는지 | 그 판단의 기술적 깊이 |
| 독자 | 채용 담당자 (2-3분 체류) | 기술에 관심 있는 사람 |
| 톤 | 의사결정 보고서 | 기술 에세이/블로그 |
| 포함 | 문제, 대안 비교, 채택 근거, 결과 | 코드, 구현 디테일, 벤치마크, 회고 |

**절대 하지 말 것:** 인사이트에 이미 있는 구현 상세(코드 블록, 상세 구현 과정)를 작업물에 복사하지 않는다. 대신 인사이트 링크로 유도한다.

## 작업물 콘텐츠 구조

아래 구조를 따라 마크다운으로 작성한다. 모든 섹션이 필수는 아니며, 프로젝트 특성에 맞게 조절한다.

### 1. 프로젝트 개요 및 역할

2-3문장으로 프로젝트의 맥락과 본인의 역할을 설명한다.
- 무엇을 만들었는지 (서비스의 목적)
- 어떤 제약/환경에서 작업했는지
- 본인이 담당한 범위

### 2. 핵심 챌린지 및 문제 해결

프로젝트당 2-3개의 챌린지. 각 챌린지는 아래 구조를 따른다.

#### 문제 상황
- 구체적 증상을 먼저 서술 (추상적 설명 금지)
- 가능하면 번호 매긴 리스트로 문제를 분리
- 정량 데이터가 있으면 포함 (응답 시간, 건수, 에러율 등)

**좋은 예:** "수백만 건의 로그가 적재되면서 DB GUI 툴이 다운되는 현상 발생"
**나쁜 예:** "로깅에 문제가 있었습니다"

#### 고려한 대안과 판단

**반드시 테이블 형태로 작성한다.** 채용 담당자가 한눈에 비교할 수 있어야 한다.

```markdown
| 대안 | 접근 | 기각 사유 |
|------|------|-----------|
| 대안 이름 | 한 줄 설명 | 왜 안 됐는지 |
| **채택한 대안** | 한 줄 설명 | — |
```

테이블 아래에 채택 근거를 1-2문장으로 요약한다.

이 섹션이 중요한 이유: 채용 담당자는 "정답을 아는 사람"보다 "왜 그 답을 선택했는지 설명할 수 있는 사람"을 찾는다. 대안 비교가 없으면 "그냥 아는 거 했다"로 읽힌다.

#### 해결 과정
- 불릿 포인트로 핵심 접근만 서술 (3-5개)
- **구현 디테일과 코드는 넣지 않는다** — 작업물의 역할이 아님
- 대신 "무엇을 했고 왜 그렇게 했는지"에 집중

#### 인사이트 링크 (해당하는 경우)
관련 인사이트가 있으면 해결 과정 아래에 인용구로 링크를 넣는다.

```markdown
> 이 챌린지의 기술적 상세는 인사이트 **[글 제목](/insights/{slug})**에서 다루고 있습니다.
```

#### 결과
- 불릿 포인트로 정량 결과 위주 (Before→After 수치)
- 정량화가 어려우면 정성적 결과라도 구체적으로

**좋은 예:** "파일 I/O 95% 감소: 요청당 1회 → 20건당 1회"
**나쁜 예:** "성능이 개선되었습니다"

### 3. 프로젝트 회고

"지금 다시 만든다면" 관점에서 1-2가지 개선 포인트를 불릿으로 간결하게 작성한다. 기술적 성장을 보여주는 섹션이지만, 길어지면 안 된다. 각 포인트는 2-3문장으로 제한.

## 작성 절차

1. **기존 콘텐츠 확인:** 해당 Feature의 현재 `content` 필드를 읽는다
2. **연관 인사이트 확인:** 해당 Feature에 연결된 인사이트(`featureSlug` 매칭)를 확인한다
3. **인사이트 내용 참조:** 연관 인사이트가 있으면 읽어서 기술적 맥락을 파악하되, 내용을 그대로 옮기지 않는다
4. **위 구조에 맞게 작성:** 의사결정 중심으로 재구성
5. **인사이트 링크 삽입:** 구현 상세가 인사이트에 있으면 링크로 유도
6. **seed 데이터 수정:** `next/prisma/data/features.ts`의 해당 항목 `content` 필드를 업데이트
7. **seed 실행:** `npx prisma db seed`로 DB 반영

## 톤 가이드

- 경어체 사용 (했습니다, 설계했습니다)
- 기술 용어는 자연스럽게 사용하되 약어는 풀어쓰기 (FTS → Full-Text Search(FTS))
- "단순 구현이 아니라 판단을 했다"는 느낌이 글 전체에서 전달되어야 한다
- 과도한 자기 PR 금지 — 팩트와 수치로 말하게 한다

## 분량 가이드

- 전체 content: 2000-3500자 (한글 기준)
- 챌린지당: 500-1000자
- 회고: 200-400자

## 파일 위치

- Feature seed 데이터: `next/prisma/data/features.ts`
- Insight seed 데이터: `next/prisma/data/insights.ts`
- Feature DTO: `next/src/core/application/dtos/feature.dto.ts`
